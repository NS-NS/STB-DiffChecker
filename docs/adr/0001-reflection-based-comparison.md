# 0001: リフレクションによる汎用比較エンジン

- ステータス: 採用（元設計を維持する判断。2026-07）
- 関連: [0003](0003-version-as-assembly.md), [0006](0006-wasm-aot-trimming.md)

## 文脈
ST-Bridgeは600要素超×数十属性のスキーマを持ち、バージョン改定で頻繁に増減する。
属性ごとに比較コードを手書きするのは非現実的。元設計は「XSDから自動生成したクラスを
実行時リフレクションで走査し、存在する属性を全部比較する」方式を採っていた。

## 決定
リフレクション方式を維持する。性能問題には以下で対処する:
- 型ごとのリフレクション情報キャッシュ(ObjectComparer.TypeShape)
- 節点id検索の辞書化(StbNodeFinder.FindNode)
- 配列マッチングの型バケット化
- Web版はAOTコンパイル([0006](0006-wasm-aot-trimming.md))

## 理由
- 「スキーマにある情報は全部比較する(比較漏れしない)」「クラス再生成だけで新バージョンに追従」
  という製品の核となる性質がこの方式から来ている
- 上記の最適化でデスクトップは実用上十分速い(実データ0.7秒程度)

## 再検討の条件
Web版を実務の主戦場にする等でさらに速度が必要になった場合、
**ビルド時コード生成(Source Generator等)への移行**が正統進化となる。
ポリシー(全属性比較・再生成追従)とICompare/IProperty資産を保ったまま、
リフレクションのコストをビルド時に移せる。実行時リフレクションはWASM環境で
特に遅い(AOTの恩恵が薄い)ことが移行動機になる。

# 0005: Web版はBlazor WebAssembly + GitHub Pages（データ非送信）

- ステータス: 採用（2026-07）

## 文脈
「GitHub Pages上で比較が完結する」ことが要件。stbファイルは構造設計情報であり、
外部サーバーへ送信しない設計が望ましい。比較ロジックはC#資産(DiffCheckerCore+STBridge2XX)がある。

## 決定
- Blazor WebAssemblyを採用し、**比較をすべてブラウザ内で実行**する
  (デスクトップと同一のDLLを使用。サーバー処理なし=静的ホスティングのGitHub Pagesで配信可能)
- ファイルはアップロードという工程自体が存在しない。CSP(connect-src 'self'等)で
  外部通信をブラウザレベルでも遮断する
- JavaScriptはCSVダウンロードの1関数のみ(js/app.js)。UIはRazor(C#)で書く

## 理由
- JS/TSでの比較ロジック再実装(二重管理)を回避できる唯一の選択肢だった
- データ非送信は設計情報を扱う上でのセールスポイントになる

## 影響・制約
- 初回ロードで.NETランタイム+DLL数十MBのダウンロードが発生(2回目以降はキャッシュ)
- シングルスレッドのため、重い処理はタブ単位に分割して進捗表示しつつUIに制御を返す
- GitHub PagesはHTTP応答ヘッダを設定できない
  → WASMマルチスレッド(要COOP/COEP)は使えない。CSPは<meta>タグで設定
- UIの検索等は再描画コストを意識する(検索はEnter確定時のみ実行。入力中は処理ゼロ)
